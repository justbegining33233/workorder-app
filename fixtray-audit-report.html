Project Overview & Tech Stack

FixTray is a multi-tenant auto-shop work order management system. It lets shops create work orders, manage customers and technicians, track appointments, handle payments, and communicate in real-time. It is built as a modern web application that can also run on phones as a native app.

💡

What does "multi-tenant" mean?

Multiple different auto shops can each have their own account. They all share the same application code and database, but their data is kept completely separate — like apartments in the same building, each with its own locked door.

Authentication & Security

💡

Authentication basics

Authentication is how the app knows who you are. FixTray uses JWT tokens (like a signed hall pass — "this person is allowed in, signed by the server") and bcrypt password hashing (passwords are scrambled into unreadable text before storage so even if the database is stolen, passwords can't be read).

src/lib/auth.ts — Core Authentication

✓ Well Built

The main auth file is well written and follows best practices:

  • JWT tokens expire in 24 hours — short-lived, so stolen tokens quickly become useless
  • Refresh tokens expire in 30 days — stored in secure, HTTP-only cookies (can't be stolen by JavaScript)
  • 2FA tokens expire in 5 minutes — strict time limit for two-factor authentication
  • bcrypt with 12 rounds — industry-standard password hashing strength
  • The JWT secret is loaded lazily — it won't crash the app during the build process

Two-Factor Authentication (2FA / TOTP)

✓ Implemented

The app has time-based one-time password (TOTP) 2FA — the same system used by Google Authenticator. 2FA secrets are encrypted with AES-256-GCM (military-grade encryption) before being stored in the database. This is excellent security practice.

CSRF Protection

✓ Implemented

CSRF (Cross-Site Request Forgery) attacks try to trick a logged-in user into taking actions they didn't intend. FixTray uses a "double-submit cookie" pattern to prevent this. The implementation is correct.

Rate Limiting

✓ Implemented

Rate limiting prevents brute-force attacks (where a hacker tries thousands of password combinations). There are actually two rate limiting systems:

  • rate-limit.ts — the older middleware-style system
  • rateLimit.ts — the newer one used by auth routes

Both use Redis (a fast in-memory store) when available, with an in-memory fallback. The duplication is messy but both work.

🔧 Recommendation: Consolidate to one rate limiter to reduce confusion.

Password Encryption in Database

✓ Automatic

The Prisma client (the database connector) automatically hashes passwords before saving. If a plain-text password somehow gets sent to the database, it gets hashed automatically. Bank account info is automatically encrypted with AES-256-GCM. This is a great safety net.

src/lib/db.ts — SQL Injection File Deleted

✓ Fixed (Session 2)

The src/lib/db.ts file, which used Prisma.raw() string interpolation (a SQL injection vector), has been deleted. It was confirmed to be dead code — imported by zero files — before removal. The threat has been fully eliminated.

.env.example File

✓ Present

A comprehensive .env.example file exists and covers all 30+ environment variables grouped by category: Database, Authentication, Stripe, Email (Resend), SMS (Twilio), Web Push (VAPID), Redis/Upstash, Sentry/OpenTelemetry, Cloudinary, Capacitor, and feature flags. Any new developer can get started immediately.

src/middleware.ts — Server-Side Route Protection

✓ Created (Session 2)

A Next.js Edge Middleware file (src/middleware.ts) has been created. This is the most important security addition of this audit session.

What it does: Every page request is intercepted before Next.js renders it. The middleware:

  • Verifies the JWT from the sos_auth cookie using jose (Edge-compatible JWT library)
  • Redirects unauthenticated users to /auth/login?redirect=...
  • Redirects authenticated users who visit a wrong-role page to their own home page
  • Runs before any React code executes — so no flash of unauthorized content

In plain English: Previously a security guard only checked IDs after you were already inside. Now there's a guard at the door. API routes still do their own checks too (defense in depth).

The middleware uses jose (installed this session) instead of jsonwebtoken because Next.js Edge Runtime does not have access to Node.js built-in modules. jose uses the Web Crypto API which works everywhere.

What are TypeScript type stubs?

TypeScript needs to know the "shape" of every package it uses — what functions exist, what parameters they take. A .d.ts type stub is like a fake blueprint that says "yes, this package exists and it looks like this" without the package actually being installed. It tricks the compiler into staying quiet.

Test Infrastructure

Almost No Tests Exist

🔴 Critical Gap

The project has Jest configured for unit tests and the package.json includes scripts for both unit and end-to-end tests. However:

  • There is only 1 test file in the entire project: src/lib/apiVersioning.test.ts
  • There are zero tests for authentication, work orders, payments, or any critical business logic
  • There is no Playwright config (playwright.config.ts), but there's a test:e2e script that tries to run Playwright — it would fail immediately
🚨

Why does this matter?

Without tests, you can't refactor code safely, you can't catch regressions (old bugs coming back), and you can't be confident that a code change didn't break something else. For a billing and customer-data application, this is a significant risk.

🔧 Action: Write tests for at minimum: login/logout, work order CRUD, and Stripe webhook handling. Create a playwright.config.ts if e2e tests are desired.

Socket.IO Incompatible with Vercel

⚠️ Important

Vercel runs each API route as a separate serverless function — they start and stop instantly and can't maintain a persistent connection. Socket.IO requires a persistent server. If deployed to Vercel, all real-time features will silently fail. The NEXT_PUBLIC_ENABLE_REALTIME flag should be set to false for Vercel deployments.
